## L2-1：AIセキュリティとは何か

### AIセキュリティの定義

**AIセキュリティとは、AIを安全に利用し、開発し、運用するための考え方**です。

守るべき対象はAIモデルだけではありません。以下を含む **AIに関わる仕組み全体** が対象です。

- AIに渡す入力データ
- AIが生成する出力
- プロンプト（システムプロンプトを含む）
- 外部ツール・外部API
- 利用者
- 業務システム・ログ

### なぜ重要なのか

生成AIは業務効率化に役立つ一方で、これまでのシステムにはなかった新しいリスクを生み出します。

- 利用者が個人情報や社内機密をAIに入力してしまう
- AIが事実ではない内容をもっともらしく回答する（ハルシネーション）
- 攻撃者が不正な指示でAIのルールを無視させようとする（プロンプトインジェクション）
- RAGが権限のない社内文書を検索し、回答に含めてしまう
- AIエージェントの誤判断が、メール送信やデータ更新などの実操作につながる

> **ポイント**：AI活用の範囲が広がるほど、AIセキュリティの重要性は高くなる。

### AIは「便利な道具」だが「完全ではない」

- 間違った内容を回答することがある
- 知らないことでも回答を作ってしまう
- 根拠のない情報を自信があるように説明する
- 外部から与えられた文書（不審な指示を含む場合も）の影響を受ける
- 生成する文章やコードが常に安全とは限らない

この特徴を理解することが **AIセキュリティの出発点** です。

### 「入力」「出力」「実行」の3つで整理する

| 観点 | 確認すること |
|------|------------|
| **入力** | AIにどんな情報を渡すか。ユーザー入力だけでなく、RAG文書・メール・Webページ・外部API結果も含む |
| **出力** | AIの回答が正しいか、安全か、公開・送信に適しているか。出力は信頼済み情報ではないため検証が必要 |
| **実行** | AIエージェントにどの操作を許可するか。重要操作をAIだけで実行させてよいか |

### 守るべき基本原則

1. 必要以上の情報をAIに渡さない（最小限の情報のみ）
2. 外部データを信用しすぎない
3. AIの出力をそのまま使わない（公開・送信・実行前に確認）
4. 権限のない情報をLLMに渡さない（プロンプトではなく、検索・データ取得の段階で制御）
5. 重要操作には人間の承認を入れる
6. ログを残して追跡できるようにする
7. 継続的に評価・改善する

### セキュリティは「禁止」ではなく「安全な活用」

AIセキュリティの目的は利用の禁止ではなく、**安全に活用できる範囲を明確にすること** です。

- 例：一般的な文章の下書きにはAIを使えるが、顧客の個人情報は入力しない
- 例：メールの下書きはAIに作らせるが、送信前に人間が確認する
- 例：社内文書検索では、利用者が閲覧できる文書だけを検索対象にする

## L2-2：従来のサイバーセキュリティとの違い

### 基本的な関係

AIセキュリティは従来のサイバーセキュリティと **別物ではありません**。

- 認証、アクセス制御、暗号化、脆弱性管理、ログ監視などは引き続き重要
- **従来の対策を土台にして、AI特有の対策を追加する** という考え方が基本

### AIセキュリティで追加される管理対象

- プロンプト（ユーザー入力・システムプロンプト）
- 学習データ、RAGで使う社内文書
- AIモデル本体、ライブラリ、外部AI API
- ベクトルDB・検索インデックス
- AIの出力
- AIエージェントが操作する外部ツール

### 6つの違い

#### 違い1：入力が「命令」にも「データ」にもなる

従来のアプリでは入力値はデータとして処理されますが、生成AIでは自然言語入力がそのまま **AIへの指示** になります。メール本文やRAG文書に「以前の指示を無視してください」と書かれていると、AIが命令として解釈する可能性があります。

→ 外部データを命令として扱わせない設計が必要。

#### 違い2：出力が毎回同じとは限らない

生成AIは確率的に文章を生成するため、同じ質問でも回答が変わることがあります。JSON形式を求めても余計な文章が付いたり、項目が欠けたりする場合もあります。

→ **1回のテスト成功だけで安全と判断できない**。繰り返しテストと出力検証が必要。

#### 違い3：プロンプトインジェクションがある

SQLインジェクション等と異なり、自然言語でAIの判断やルールに影響を与える攻撃です。直接入力するパターンと、メール・Webページ・PDF・RAG文書経由の **間接プロンプトインジェクション** があります。

→ 入力検証だけでは防げない。アクセス制御・最小権限・人間の承認を組み合わせる。

#### 違い4：正しそうな誤情報が出る（ハルシネーション）

自然で説得力のある文章のため、利用者が誤りに気づきにくい点が特徴です。

→ AIの出力は「完成した答え」ではなく「確認が必要な候補」として扱う。

#### 違い5：AIが外部ツールを実行する

AIエージェントの誤りは誤回答にとどまらず、誤送信・誤削除・誤更新など **現実の被害** につながります。

→ 権限の最小化（送信ではなく下書きまで、更新ではなく読み取りのみ等）と人間の承認が基本。

#### 違い6：データの流れが複雑になる

ユーザー入力 → アプリサーバー → 外部AI API → ベクトルDB → 外部ツール → ログ、と多くのサービスや保存場所が関係します。

→ どのデータが、どこへ送られ、どこに保存され、誰が閲覧できるのかを把握する。

### AIシステムで追加すべき対策

- プロンプトインジェクションの影響を前提にした設計
- 外部データは「命令」ではなく「参考情報」として扱う
- 出力の正確性・安全性・形式の確認
- RAGの権限に応じた検索制御
- 個人情報・機密情報のマスキングと最小化
- AIエージェントへの最小権限付与と重要操作への人間の承認
- モデル・API・ライブラリの出所と更新の管理
- リクエスト数・トークン数・コストの監視

## L2-3：生成AI時代に増えたリスク

### リスクが増えた理由

**AIが扱うデータと操作の範囲が広がった** ためです。自然言語で簡単に情報を入力でき、外部AI API・RAG・AIエージェント・ログなど、関係する場所が増えました。

### 9つの代表的リスク

| # | リスク | 概要 | 主な対策 |
|---|--------|------|---------|
| 1 | **情報の入力しすぎ** | 個人情報・機密情報・認証情報を深く考えずに入力してしまう | 入力前の確認、マスキング、必要最小限の入力 |
| 2 | **もっともらしい誤情報** | 存在しない制度・出典、誤った数値を自然な文章で生成 | 元資料・公式情報・専門家による確認 |
| 3 | **プロンプトインジェクション** | 不正な指示でAIのルールや動作を変えようとする攻撃 | プロンプトでの禁止だけに頼らず、アプリ側で制御 |
| 4 | **外部データによる影響** | Webページ・PDF・メール・RAG文書経由でAIの判断が影響を受ける | 外部データを参考情報として扱い、それだけで判断・操作しない |
| 5 | **RAGによる情報漏洩** | アクセス制御の誤りで権限外の文書が検索・回答される | 文書分類、検索前のアクセス制御、根拠表示、削除管理 |
| 6 | **AI出力の不安全な利用** | 生成されたHTML・SQL・コマンド・コードを検証せず利用 | 形式・内容・安全性を検証してから利用 |
| 7 | **AIエージェントの誤操作** | 誤判断が誤送信・誤共有・誤更新などの実操作につながる | 最小権限、重要操作への人間の確認・承認 |
| 8 | **外部依存とサプライチェーン** | モデル・API・ライブラリ等の脆弱性・改ざん・停止の影響 | 提供元・ライセンス・バージョン・脆弱性の継続管理 |
| 9 | **可用性とコスト** | 大量リクエストやエージェントの繰り返し処理による負荷・費用急増 | リクエスト数・トークン数・実行時間・ツール呼び出し回数の上限設定と監視 |

> **ポイント**：AIが扱うデータ、AIの出力、AIに許可する操作の範囲を管理すること。便利さだけでなく「どのような失敗が起こり得るか」を考えて設計する。

## L2-4：使う側・作る側・管理する側のリスク

### 3つの立場

| 立場 | 役割 |
|------|------|
| **使う側** | AIに質問・情報を入力し、回答や生成物を業務で利用する |
| **作る側** | AIを組み込んだアプリ・業務システムを設計・実装する |
| **管理する側** | 組織として利用ルールを作り、契約・運用・監視を管理する |

どれか1つの立場だけが注意しても、AIセキュリティは成立しません。

### 使う側

**主なリスク**

- 個人情報・機密情報の入力
- 誤回答を信じてしまう
- 出力を未確認のまま送信・公開する
- 生成コードを未確認で実行する
- 著作権・契約・社内ルール上の問題の見落とし

**気をつけること** — 基本姿勢は「**入力前に立ち止まり、出力後に確認する**」

- 入力前：個人情報・機密・認証情報を含まないか確認。必要なら仮の情報に置き換える
- 出力後：数値・日付・出典を照合、公開・送信前にレビュー、コードは検証環境でテスト
- 組織のAI利用ルールに合っているか確認

### 作る側

**主なリスク**

- プロンプトインジェクションの未想定
- システムプロンプトへの秘密情報の記載
- 外部AI APIへの機密情報の過剰送信
- RAGのアクセス制御の失敗
- 出力の未検証利用（HTML・SQL・コマンド等）
- AIエージェントへの過剰な権限付与
- ログへの情報の残しすぎ

**気をつけること** — 「**入力・データ・出力・実行をそれぞれ制御する**」

- 入力のサイズ・形式を検証（ただし入力検証だけに依存しない）
- 外部データは命令ではなく参考情報として扱う
- 権限のない情報は検索結果にもLLM入力にも含めない
- 出力は後続処理に渡す前に形式・値・安全性を検証
- エージェントには最小権限、重要操作には人間の承認
- ログの保存内容・閲覧権限・削除時期を設計

### 管理する側

**主なリスク**

- 利用ルールの曖昧さによる判断のばらつき
- 未承認サービスの利用（シャドーAI）
- 外部APIの契約・データ利用条件の未確認
- RAG取り込み基準の欠如
- 利用状況・コストの未監視
- インシデント対応手順の欠如

**気をつけること** — 「**組織としてのルールと運用体制を整える**」

- 利用可能なサービス・モデルの明確化（データ保存・学習利用・削除・利用地域の確認）
- 入力禁止情報の定義（認証情報、未公開情報、顧客の個人情報など)
- 出力のレビュールール、RAG取り込み基準、エージェント許可操作の決定
- 重要操作の承認者の明確化
- ログの保存範囲・閲覧権限・保存期間の設計
- インシデント対応手順（報告先・停止・影響調査・復旧）の準備

### 共通して重要な考え方

1. AIを過信しない
2. 必要以上の情報をAIへ渡さない
3. 外部データを信頼済みの情報・命令として扱わない
4. AIの出力は確認してから使う
5. 権限は必要最小限に
6. 重要操作には人間の承認を入れる
7. 追跡可能なログを残す
8. ルールや対策を継続的に見直す

## L2-5：代表的なAIセキュリティフレームワーク

### フレームワークを使う目的

- AIリスクを **体系的に整理** し、対策の抜け漏れを減らす
- 開発者・セキュリティ担当者・管理者・経営層が **共通の用語** で話せるようにする
- チェックリスト、リスク評価、監査、運用ルール作成に活用する

> **注意**：項目を確認するだけで安全になるわけではない。自社のデータ・利用者・業務・権限・システム構成に合わせて具体的な対策へ落とし込むことが重要。

### 主要フレームワーク一覧

| フレームワーク | 目的・特徴 | 向いている用途 |
|--------------|-----------|--------------|
| **NIST AI RMF** | AIリスクを組織的に管理する汎用フレームワーク。Govern / Map / Measure / Manage の4機能 | AIリスク管理の全体像の整理 |
| **NIST Generative AI Profile** | AI RMFを生成AI向けに具体化。ハルシネーション、情報漏洩、著作権、バリューチェーン等を扱う | 生成AI特有のリスクの詳細確認 |
| **OWASP Top 10 for LLM Applications** | LLMアプリの重要リスクを整理（Prompt Injection、Sensitive Information Disclosure、Supply Chain、Improper Output Handling、Excessive Agency 等） | LLMアプリ開発時の具体的チェックリスト |
| **OWASP Top 10 for Agentic Applications** | AIエージェント特有のリスク（権限・実行）を整理 | エージェントの設計・評価 |
| **MITRE ATLAS** | AIシステムへの攻撃者の戦術・技術の知識ベース | 脅威モデリング、レッドチーミング、攻撃シナリオ分析 |
| **ISO/IEC 42001** | AIマネジメントシステムの国際規格。方針・責任体制・ライフサイクル管理・継続的改善 | 組織的なAIガバナンスの構築 |

### NIST AI RMF の4つの機能

| 機能 | 内容 |
|------|------|
| **Govern** | 方針、役割、責任、組織体制を整える |
| **Map** | AIの利用目的、環境、利用者、影響を受ける人を把握する |
| **Measure** | 性能、安全性、公平性、セキュリティを評価する |
| **Manage** | リスクに優先順位を付け、対策を実施する |

### 実務での使い方（組み合わせの例）

1. **NIST AI RMF** で組織全体のリスク管理方針を整理
2. **OWASP Top 10** でアプリ・エージェントの具体的な脆弱性を確認
3. **MITRE ATLAS** で攻撃シナリオを洗い出し
4. **ISO/IEC 42001** を参考に責任体制・承認手順・監査・継続的改善を整備
5. 自社の業務に合わせてチェックリスト・開発基準へ落とし込む

### 利用時の注意点

- フレームワークは完全な答えではない。自社の状況に合わせて調整する
- AI技術・攻撃手法は変化するため、更新状況を確認する
- 法令・契約・業界規制・社内ルールは別途確認が必要
- チェックリスト作成で終わらせず、実装・運用まで行う
- 定期的に見直す

## L2-6：リスクマップで見るAIシステム

### リスクマップとは

AIシステムのリスクは複数の場所に分散しています。**処理の流れに沿って「どこで、どのような問題が起こり得るか」を可視化** するのがリスクマップです。

```
ユーザー → 入力 → アプリ（プロンプト作成）→ LLM / 外部AI API
                          ↑
                    RAG（文書検索）
                          ↓
          出力 → 画面表示 / メール / 自動処理
                          ↓
          ツール実行（エージェントの場合）
                          ↓
                  ログ・監視基盤
```

### 確認地点ごとのリスクと対策

#### 1. 入力地点

**リスク**：個人情報・機密情報・認証情報の入力、プロンプトインジェクション目的の指示、過大な入力による負荷・コスト増、不正なファイル

**対策**：入力してよい情報・文字数・ファイル形式・サイズの制限、警告・検証

#### 2. プロンプト・LLM周辺

**リスク**：システムプロンプトの推測・流出（秘密情報を記載していると影響大）、プロンプトインジェクション、外部AI APIへの機密情報送信、ハルシネーション、モデル更新による挙動変化

**対策**：システムプロンプトに秘密情報を入れない、送信データの確認、プロンプトとモデル挙動の継続評価

#### 3. RAG・外部データ

**リスク**：権限外文書の検索、古い文書による誤回答、文書内の悪意ある指示（間接プロンプトインジェクション）、メタデータからの情報漏洩、削除済み文書がベクトルDBに残存

**対策**：取り込み・検索・表示・削除まで含めた管理、利用者権限に応じた検索制御

#### 4. 出力地点

**リスク**：誤情報、個人情報・機密情報の混入、不適切表現、脆弱なコード、危険なコマンド、想定外の形式

**対策**：表示・公開・送信・実行の前に、内容・形式・安全性を検証

#### 5. ツール実行（AIエージェント）

**リスク**：誤送信による情報漏洩、機密ファイルの誤共有、誤削除・誤更新、権限設定の誤変更、外部APIへの過剰な情報送信、本番環境での危険なコード実行

**対策**：最小権限、重要操作（外部送信・削除・更新・本番操作）への人間の承認

#### 6. ログ・監視

**リスク**：質問・会話履歴・参照文書・回答がログに残り、ログ自体が情報の集まりになる。エラー内容に内部パスや設定情報が含まれる

**対策**：記録する内容・閲覧権限・保存期間の設計、認証情報や文書全文は保存しないかマスキング

### AIシステムのリスク確認チェックリスト

- [ ] AIへ入力してよい情報を制限しているか
- [ ] 外部AI APIへどのデータを送信しているか把握しているか
- [ ] RAGの検索対象に利用者ごとのアクセス制御があるか
- [ ] 外部データ（Webページ・メール・文書）をAIへの命令として扱っていないか
- [ ] AIの出力を、公開・送信・実行の前に検証しているか
- [ ] ツール実行に最小権限を適用し、重要操作に人間の承認を入れているか
- [ ] ログに個人情報や機密情報を残しすぎていないか

## セクション全体のまとめ

- **AIセキュリティ** とは、AIを安全に利用・開発・運用するための考え方。モデルだけでなく、入力・出力・データ・権限・外部ツール・運用を含む **AIアプリ全体** を対象とする
- **従来のサイバーセキュリティが土台**。その上に、プロンプトインジェクション対策、出力検証、RAGのアクセス制御などAI特有の対策を追加する
- 生成AI時代のリスクは、**AIが扱うデータと操作の範囲が広がった** ことで増えた。「入力」「出力」「実行」に分けて整理すると分かりやすい
- **使う側・作る側・管理する側** がそれぞれの責任を果たし、連携することが不可欠
- **フレームワーク**（NIST AI RMF、OWASP Top 10、MITRE ATLAS、ISO/IEC 42001 等）は目的に応じて組み合わせ、自社の設計・開発・運用へ反映する
- **リスクマップ** でデータの流れと操作範囲を可視化し、各地点のリスクと対策を対応させることで、抜け漏れを減らせる

### キーワード一覧

| 用語 | 意味 |
|------|------|
| ハルシネーション | AIが事実ではない内容をもっともらしく生成すること |
| プロンプトインジェクション | 不正な指示によりAIの本来のルールや動作を変えようとする攻撃 |
| 間接プロンプトインジェクション | Webページ・PDF・メール・RAG文書などに埋め込まれた指示を通じてAIが影響を受けること |
| RAG | 質問に関連する文書を検索し、その内容をLLMへ渡して回答させる仕組み |
| AIエージェント | AIの判断をもとに外部ツール（メール・ファイル・DB等）を実行する仕組み |
| 最小権限 | 目的達成に必要な最小限の権限だけを与える原則 |
| 不安全な出力処理 | AIの出力を検証せずHTML・SQL・コマンド等として利用する問題 |
| AIサプライチェーン | モデル・データ・ライブラリ・外部API等、AIアプリが依存する外部要素の連なり |
