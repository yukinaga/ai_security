## AI利用で起きる情報漏洩とは

### 情報漏洩は「AIが秘密を話す」ことだけではない

AI利用時の情報漏洩は、**入力・出力・保存・共有・連携の全体** で考える必要があります。情報がどこから入り、どこに保存され、どこへ出ていくのかを意識することが基本です。

### 情報漏洩が起きる主な場面

| 場面 | 何が起こるか |
|------|------------|
| **入力時** | 顧客情報・社内資料・APIキー入りのコードなどをそのまま入力。入力した時点で外部サービスやログに送られる可能性がある |
| **出力時** | 本来出してはいけない情報（他部署の機密、個人名、システムプロンプト等）が回答に含まれる |
| **保存・ログ** | 入力や出力がログ・会話履歴に残り、ログ自体が機密情報の集まりになる |
| **共有時** | AIが生成したメール・提案書・要約に社内情報が混ざったまま外部共有。共有リンクの権限設定ミスも |
| **外部API利用時** | ユーザー入力に加え、会話履歴・システムプロンプト・RAG文書なども外部サービスへ送信される |
| **RAG利用時** | 閲覧権限のない文書が検索される、回答に文書内容を出しすぎる、検索ログに機密が残る |
| **AIエージェント利用時** | 宛先間違いのメール送信、ファイルの誤共有、機密情報のチャット投稿など、**漏洩が実際の操作として起こる** |

### 特に注意が必要な情報

個人情報（氏名・住所・連絡先・社員番号・顧客ID）、顧客情報・社員情報、契約情報・財務情報・未公開の事業計画、ソースコード・APIキー・パスワード・アクセストークン、社内規程・内部メモ、医療・金融・法律に関わる情報。

### 情報漏洩を防ぐ基本方針

1. AIに入力する前に情報を確認する（外部に送ってよいか、社内ルール上問題ないか）
2. 必要な情報だけをAIに渡す
3. 個人情報・機密情報は必要に応じてマスキングする
4. AIに見せる文書を制限する（RAGは権限に応じて検索対象を分ける）
5. AIの出力は共有前に確認する
6. ログに残す情報も制限する
7. 重要操作には人間の確認を入れる

> **ポイント**：AIに渡す情報と、AIから出る情報の **両方を管理する**。

## 入力してはいけない情報の例

### 基本方針

一度入力した情報は、AIサービス、ログ、会話履歴、APIリクエスト、共有先などに残る可能性があります。**「入力する前に確認する」** こと、そして **「必要最小限の情報だけを渡す」** ことが基本です。

### 注意すべき情報の種類

| 種類 | 例 | 注意点 |
|------|-----|--------|
| **個人情報** | 氏名、住所、電話番号、メールアドレス、生年月日、顔写真、マイナンバー、顧客ID、社員番号 | 名前を消しても、部署名・役職・日付等の組み合わせで個人を推測できる場合がある |
| **顧客情報** | 顧客名、商談メモ、購入履歴、契約条件、見積金額、クレーム内容 | 会社の信用や契約上の責任に関わる。NDA対象の資料はAI入力が契約違反になる可能性も |
| **社員情報・人事情報** | 社員名簿、評価コメント、給与情報、勤怠情報、健康情報、採用候補者情報、面接メモ | 個人情報の中でも特にセンシティブ。本人のプライバシーやキャリアに直結する |
| **機密情報・社外秘情報** | 未公開の事業計画、新製品情報、財務情報、営業戦略、価格戦略、役員会資料、研究開発情報 | 漏洩は競争上の不利益に直結。原則マスキング・抽象化し、必要なら社内承認を取る |
| **契約・法務関連情報** | 契約書全文、NDA内容、交渉中の契約、訴訟・紛争情報、法務相談内容 | 契約によっては第三者サービスへの送信自体が制限されている |
| **認証情報・技術的秘密** | パスワード、APIキー、アクセストークン、秘密鍵、接続文字列、認証Cookie、本番環境の設定 | **絶対に入力しない**。漏洩は不正アクセスに直結 |
| **ソースコード・システム情報** | 非公開コード、設計書、システム構成図、DB定義、エラーログ、脆弱性診断結果 | エラーログにはファイルパス・サーバー名・SQL文・接続情報が含まれることがある |
| **センシティブな業務情報** | 医療・健康情報、金融・与信情報、学生・児童の情報、内部通報、インシデント報告 | 業界の法規制により特に厳格な管理が求められる場合がある |

### 入力前にできる対策

- 個人名・会社名を置き換える（「株式会社ABC」→「A社」、「山田太郎さん」→「社員A」）
- メールアドレス・電話番号は削除、金額・ID・契約番号は必要に応じて伏せる
- 全文ではなく必要な部分だけを抜き出す
- 公開情報で目的を達成できないか考える
- 利用するAIツールの設定（保存・学習利用・管理者閲覧）を確認する

> **迷った場合の基本**：入力しない、マスキングする、担当者に確認する。

## 個人情報・機密情報・認証情報の違い

### 3つの情報の比較

| | 個人情報 | 機密情報 | 認証情報 |
|---|---------|---------|---------|
| **定義** | 特定の個人を識別できる情報 | 組織や業務上守るべき情報 | システムにアクセスするための情報 |
| **例** | 氏名、住所、メールアドレス、社員番号、顧客ID | 契約書、事業計画、営業戦略、財務情報、研究開発情報 | パスワード、APIキー、アクセストークン、秘密鍵 |
| **守る理由** | 個人のプライバシーを守るため | 会社の信用・競争力・契約関係を守るため | 不正アクセスを防ぐため |
| **AI入力時の扱い** | **削除・匿名化・マスキング** | **公開範囲と社内ルールを確認** | **絶対に入力しない** |

### 個人情報の注意点：再識別のリスク

単体では個人を特定できなくても、**他の情報と組み合わせると特定できる** 場合があります。

> 例：「東京支社の営業部長で、先週A社との商談を担当した人」——名前がなくても社内では誰のことか分かる。

→ マスキングは氏名やメールアドレスだけでなく、**文脈全体を確認** する必要があります。

### 機密情報の注意点：ルールの確認

- 社外秘か、部外秘（社内でも特定の部署限定）か
- 取引先から受領した契約上の秘密情報か（第三者サービスへの送信制限がある場合も）
- 社内規程でAIツールへの入力が許可されているか

> 「社内にある資料だから大丈夫」ではなく、「**この情報を外部AIサービスに送ってよいのか**」という観点で確認する。公開範囲が不明な情報は入力しないのが基本。

### 認証情報の注意点：漏れたら無効化と再発行

認証情報が漏れると、不正ログイン、APIの不正利用（高額請求）、データの閲覧・改ざん、権限の乗っ取り、他システムへの侵入の足がかりになります。

> **対応の基本**：見つけたら削除。漏れた可能性があれば **無効化して再発行**（マスキングすれば済む話ではない）。

### 3つが重なるケース

1つの情報が複数の分類に当てはまることがあります。

- 顧客別の契約書 = 個人情報（担当者名）+ 機密情報（契約条件）+ 契約上の秘密情報
- ソースコード内のAPIキー = 機密情報 + 認証情報
- 社員評価資料 = 個人情報 + 社内の機密情報

→ 分類が重なるほど慎重に。迷ったら **より厳しい分類として扱う** のが安全。

## チャット履歴とログのリスク

### チャット履歴とログの違い

| | チャット履歴 | ログ |
|---|------------|------|
| 性質 | ユーザーから見える会話の記録 | システム側で管理される記録 |
| 含まれるもの | 質問・回答、会話の文脈、アップロードファイル、過去の依頼内容 | 利用者ID、日時、入出力、APIリクエスト/レスポンス、参照文書、実行ツール、エラー、トークン数 |
| メリット | 会話の見返し、続きからの作業 | 障害対応、不正利用の調査、セキュリティ監査 |
| リスク | 後から閲覧・共有・流出する | ログ自体が機密情報の集まりになる |

### チャット履歴のリスク

- 過去に入力した顧客情報・社内資料が履歴に残る
- 組織向けツールでは **管理者が履歴を閲覧できる** 場合がある
- 共有リンク経由で第三者に見られる、端末・アカウント共有で他人に見られる
- 会話内容のコピーを誤って外部に共有してしまう

→ チャット履歴は単なるメモではなく、**情報管理の対象** として扱う。

### ログのリスクと場面別の注意点

- **API利用時**：自社側と外部サービス側の **両方** でログが残る可能性。保存期間・閲覧者・学習利用・削除可否を確認する
- **RAG利用時**：質問、検索クエリ、検索された文書チャンク、根拠箇所などが残る。ログ経由で文書内容が見えてしまうことも
- **AIエージェント**：誰が依頼し、AIがどう判断し、何を実行し、誰が承認したかの記録は監査に必要。一方でメール全文や機密ファイルの内容をそのまま残すのは漏洩リスク

### ログ管理で起こりやすい問題

1. 何でも全文保存してしまう（ログが機密情報の集まりになる）
2. 保存期間が決まっていない
3. ログ閲覧権限が広すぎる
4. 個人情報や認証情報がログに混ざる
5. 逆に監査に必要な情報が不足し、問題発生時に追跡できない

> ログは **多すぎても少なすぎても問題**。何を残すかを意図的に設計する。

### 安全なログ設計

| 残すべき情報 | 残さない・マスキングすべき情報 |
|------------|------------------------------|
| 利用者、利用日時、操作種別 | パスワード、APIキー |
| 参照した文書ID（全文ではなく） | 個人情報 |
| 実行したツール、承認記録 | 機密文書の全文 |
| エラー種別 | 不要な会話全文 |

### 利用者が注意すべきこと

入力内容が履歴に残るか、履歴を削除できるか、管理者が閲覧できるか、共有設定は適切か、履歴保存・学習利用の設定はどうなっているか——**利用するAIツールの設定を確認してから使う**。

## API利用時のデータ送信リスク

### 前提：AI APIを使う = 外部サービスにデータを送る

ユーザーから見ると自社アプリの画面に入力しているだけでも、裏側ではその内容が外部のAIサービスに送信されている場合があります。**どの情報がどこへ送られているのかの把握** が出発点です。

### APIリクエストに含まれる情報

ユーザー入力だけではありません。

- システムプロンプト
- 過去の会話履歴（過去のやり取りに含まれる機密情報も一緒に送信される）
- アップロードされたファイルの内容
- RAGで検索された社内文書の一部（**ユーザーが直接入力していない社内情報**）
- 外部ツールの実行結果、ユーザーIDやメタデータ

### 送信前に確認すべきこと

**送信してはいけない情報**：個人情報、顧客情報・社員情報、契約情報・機密資料、ソースコード・本番環境の設定、そして特に **認証情報（APIキー・パスワード・アクセストークン）は絶対に送信しない**。

**送信先サービスの確認**：

- どの会社のサービスか、契約条件はどうか
- データがどの地域で処理されるか
- 送信データは保存されるか、保存期間はどれくらいか
- 誰が閲覧できるか、**モデル学習に使われるか**、削除できるか

**契約・社内ルールの確認**：

- 社内ルールでAI APIの利用が許可されているか
- 顧客との契約やNDAに違反しないか
- 業界規制（医療・金融・教育・法律等）に抵触しないか
- 必要に応じて管理者・法務・情報システム部門の承認を取る

### APIキーの漏洩に注意

業務データだけでなく、APIキー自体の漏洩にも注意します。

- エラー調査のために設定ファイルをそのままAIに貼り付けない（APIキーが含まれていることがある）
- ソースコードへの直書き、ブラウザ側への配置、ログ出力、GitHubへの公開を避ける
- 漏れた可能性があれば **無効化して再発行**

### 安全なデータ送信の考え方

「全部送ってAIに考えさせる」のではなく、「**目的達成に必要な情報だけを送る**」のが基本です。

1. 不要な情報を送らない
2. 個人情報はマスキング、機密情報は削除または抽象化
3. 認証情報は必ず除外
4. 会話履歴は必要以上に送らない（現在の質問に関係ない過去の会話まで送る必要があるか）
5. RAGの検索結果は絞る（全文ではなく必要なチャンクだけ）
6. 送信前に入力チェックを行う

## マスキング・匿名化・データ最小化

### 3つの手法の違い

| 手法 | 内容 | 例 |
|------|------|-----|
| **マスキング** | 特定の情報を別の表現に置き換える | 「山田太郎」→「社員A」、「株式会社ABC」→「A社」、「1,200万円」→「金額X」 |
| **匿名化** | 個人や組織を特定できない形に加工する | 「山田太郎さんは2025年4月に東京支社へ異動」→「ある社員は2025年4月に別の拠点へ異動」 |
| **データ最小化** | 目的達成に必要な最小限の情報だけを使う | 契約書全文ではなく、確認したい解約条項だけを抜き出して入力 |

**なぜ必要か**：AIに入力する情報が多いほど、漏洩時の影響も大きくなるから。AIに渡す前に「減らす・隠す・置き換える」ことで、リスクを下げられます。

### マスキングのポイント

作業に必要な意味は残しつつ、個人や会社を特定できる情報を隠します。メール文の表現を整えるだけなら、実名や実際のメールアドレスは不要なことが多いです。

商談メモ、契約書、ソースコード、ログ、設定ファイルには識別子や秘密情報が紛れ込みやすいため、特に注意します。

### 匿名化のポイント：再識別に注意

**名前を消すだけでは不十分** な場合があります。部署名、役職、勤務地、年齢、日付、取引先名、案件名、特徴的な経歴が残っていると、誰のことか推測できます。

> 例：「大阪支社の営業部長で、先月A社との大型案件を担当した人」——名前がなくても特定可能。

少人数の部署や特定の案件に関する情報は特に注意し、**文脈全体を見て** 判断します。

### データ最小化のチェック観点

- 全文を入力する必要があるか（一部の抜粋で達成できないか）
- 個人名・会社名は本当に必要か
- 金額・IDは必要か
- 過去の会話履歴をすべて送る必要があるか
- ファイル全体を渡す必要があるか

> 情報を多く渡せば精度が上がる場合もあるが、その分リスクも増える。「便利だから全部渡す」ではなく「必要な部分だけ渡す」。

### 実務での加工例

**加工前**：

> A社の田中様より、来期予算1,200万円で導入検討中。競合はB社。現行システムの不満は処理速度。

**加工後**：

> 顧客Xの担当者Yより、来期予算は金額Zで導入検討中。競合は競合企業A。現行システムの不満は処理速度。

要約や提案文の作成という目的に必要な情報は残しつつ、顧客名や金額を伏せています。

### 認証情報はマスキングではなく削除

認証情報（パスワード、APIキー、アクセストークン、秘密鍵、接続文字列、認証Cookie、ワンタイムコード）は、「一部を隠せばよい」ではなく **入力前に完全に削除** が基本です。

入力してしまった可能性がある場合：① すぐに無効化 → ② 再発行 → ③ 利用ログを確認し不正利用がないか調査 → ④ 必要に応じて影響範囲を調査。

### 入力前チェックリスト

- [ ] 個人情報を削除またはマスキングしたか
- [ ] 顧客名や会社名を置き換えたか
- [ ] 金額・ID・契約番号を伏せたか
- [ ] 認証情報を完全に削除したか
- [ ] 全文ではなく必要部分だけにしたか
- [ ] 文脈から再識別されないか
- [ ] 社内ルール上、入力してよい情報か
- [ ] 出力を共有する前に確認するか

## ミニ演習：この情報はAIに入力してよいか

### 判断は3段階で考える

| 段階 | 内容 | 例 |
|------|------|-----|
| **① 入力してよい** | 公開済みの情報、機密性が低い情報、適切にマスキング済みの情報 | 公式サイトの会社概要 |
| **② 条件付きで入力してよい** | マスキング・匿名化・社内承認・ツール確認を行えば使える可能性がある | 匿名化したアンケート、マスキング済みの契約条項 |
| **③ 入力してはいけない** | 個人情報、機密情報、認証情報、契約上送信禁止の情報 | APIキー入りのコード、社員評価コメント |

実務では「② 条件付き」が多くなります。その場合、**どの条件を満たせば安全に使えるか** を考えます。

### ケーススタディ

#### ケース1：公開済みの会社概要 → 入力してよい可能性が高い

公式サイトに掲載済み = 社外公開前提の情報。ただし、従業員数や代表者名などが **古い情報になっていないか** は確認する。

#### ケース2：顧客名入りの商談メモ → そのままは不可

「A社の田中様より、来期予算1,200万円で導入検討中。競合はB社。」には、顧客名・担当者名・予算・競合情報が含まれる。要約やメール作成が目的なら実名や金額は不要なことが多い。**マスキングして必要最小限に加工** してから使う。

#### ケース3：社員評価コメント → そのままは不可

氏名 + 人事評価情報。本人のプライバシーとキャリアに関わり、社内でも閲覧範囲が限定される情報。匿名化しても部署・役職・エピソードから **個人が推測される可能性** がある。社内ルールや承認手順を必ず確認する。

#### ケース4：匿名化したアンケート結果 → 条件付きで可

「社員A」「社員B」としていても、属性の組み合わせで再識別できないか確認する。少人数の部署は特に注意。自由記述欄の個人名・案件名も確認し、必要なら「20代の社員」→「若手社員」のように **さらにぼかす**。

#### ケース5：APIキーを含むコード → 入力不可

APIキーは認証情報。接続文字列にはDB接続先・ユーザー名・パスワードが含まれることも。**入力前に完全に削除**。入力してしまった可能性があれば、無効化・再発行・利用ログの確認を行う。

#### ケース6：契約書の一部（マスキング済み） → 条件付きで可

解約条項だけ抜き出し、会社名・金額・契約番号を伏せている = データ最小化の実践。ただし、**契約上、外部送信が許可されているか**（NDA違反にならないか）を確認する。AIの要約は下書き・整理の補助とし、重要な判断は法務など専門部署が行う。

#### ケース7：社内会議の議事録 → そのままは不可

新規事業の方針・未公開の売上見込み・担当者名を含む = 社外秘の可能性が高い。公開可能な範囲に絞り、機密部分を削除し、名前・金額をマスキングし、社内ルールの確認と必要な承認を取る。

### 入力前チェックの流れ

1. その情報を入力してよいかを考える
2. どの情報が問題になるか特定する（個人名？顧客名？金額？契約情報？認証情報？）
3. マスキングできるか考える
4. 匿名化しても再識別されないか確認する
5. 必要最小限の情報になっているか確認する
6. 社内ルールや契約に反していないか確認する
7. 担当者への確認が必要か判断する

> **迷ったら：入力しない、加工する、確認する。** AIは情報を入力した後に取り消すのが難しい場合があるからこそ、入力前に立ち止まる。

## セクション全体のまとめ

- AI利用の情報漏洩は **入力・出力・保存・共有・連携** のすべての場面で起こり得る。AIに渡す情報と、AIから出る情報の両方を管理する
- 入力してはいけない情報の代表は、個人情報、顧客情報、社員・人事情報、機密情報、契約・法務情報、認証情報、非公開ソースコード、センシティブな業務情報
- **個人情報はマスキング、機密情報は公開範囲を確認、認証情報は絶対に入力しない**——情報の種類で扱いを分けると判断しやすい
- チャット履歴とログは便利な一方、それ自体が情報漏洩の原因になる。保存内容・期間・閲覧権限を設計し、ログは多すぎず少なすぎず意図的に残す
- AI APIの利用では、ユーザー入力以外（会話履歴・RAG文書・ファイル）も外部送信されることを把握し、送信先の保存・学習利用・契約条件を確認する
- **マスキング・匿名化・データ最小化** はAIに渡す前の基本対策。ただし名前を消すだけでは再識別され得るため文脈全体を確認する。認証情報はマスキングではなく削除、漏れたら無効化と再発行
- 入力判断は「入力してよい／条件付きで可／入力してはいけない」の3段階で考え、迷ったら **入力しない・加工する・確認する**

### キーワード一覧

| 用語 | 意味 |
|------|------|
| 個人情報 | 特定の個人を識別できる情報。単体で特定できなくても組み合わせで特定できる場合がある |
| 機密情報 | 組織や業務上守るべき情報。社外秘・部外秘・契約上の秘密情報などがある |
| 認証情報 | システムやサービスにアクセスするための情報。パスワード、APIキー、アクセストークンなど |
| 再識別 | 名前等を消しても、部署・役職・日付などの文脈から個人が推測されてしまうこと |
| マスキング | 特定の情報を別の表現に置き換えること（「山田太郎」→「社員A」） |
| 匿名化 | 個人や組織を特定できない形に情報を加工すること |
| データ最小化 | 目的達成に必要な最小限の情報だけを使うこと |
| チャット履歴 | ユーザーとAIのやり取りとして保存される、ユーザーから見える記録 |
| ログ | システム側で管理される利用状況・処理内容の記録。残しすぎても不足しても問題になる |
| 無効化と再発行 | 認証情報が漏れた可能性がある場合の基本対応。マスキングや削除だけでは不十分 |
| NDA（秘密保持契約） | 取引先と交わす秘密保持の契約。対象情報のAIツールへの入力が契約違反になる場合がある |
