## AIエージェントとは何か

### 定義

**AIエージェントとは、LLMが判断し、必要に応じて外部ツールを使いながら、目的達成に向けて処理を進める仕組み** です。通常のチャットAIが「答える」ことが中心なのに対し、AIエージェントは「**行動するAI**」として、情報を探し、ツールを使い、次に何をすべきか判断します。

### 基本構造と処理ループ

```
ユーザーの依頼 → LLMが目的を理解 → 必要な作業を判断
→ ツールを実行 → 結果を確認 → 次の処理を判断 → … → 結果を返す
```

AIエージェントは一度の回答で終わらず、**観察 → 判断 → 行動 → 結果確認 → 次の判断** というループで複数ステップの作業を進めます。このループがあるからこそ、途中の判断ミスや不正な入力の影響が **後続の操作に広がる** 可能性もあります。

### 「判断」と「実行」が入ることが特徴

通常のAIなら「返信案は以下です」と回答して終わりますが、AIエージェントはメールを読み、内容を理解し、返信を考え、下書きを作成し、設定によっては送信まで行います。**LLMの出力が次の行動につながる** 点が最大の特徴です。

### 使うツールの例とRAGとの組み合わせ

Web検索、メール作成・送信、カレンダー登録、ファイル検索・操作、データベース参照、チケット作成、社内システム連携、コード実行、外部API呼び出しなど。

RAGと組み合わせると「社内文書を検索 → 内容を整理 → 関係者向けメール下書きを作成」といった使い方ができます。ただし、**RAGの検索結果に誤りや機密情報、悪意ある指示が含まれていると、その影響がエージェントの判断や行動に広がる** ため、アクセス制御・出力制御・重要操作の承認が特に重要になります。

### 便利さとリスクはセット

| 便利になる理由 | リスクが大きくなる理由 |
|--------------|---------------------|
| 複数ステップの作業を任せられる | 外部データ（メール・文書）に不審な指示が含まれ得る |
| 情報収集と文章作成をつなげられる | 判断ミスが実際の操作につながる |
| 外部ツールと連携し業務フローに組み込める | プロンプトインジェクションで意図しない送信・共有が起こり得る |
| 定型業務を効率化できる | ログや実行履歴に機密情報が残る |

> **安全設計の基本**：権限は最小限に、外部データを信用せず、重要操作は自動実行せず、ログを残し、失敗時に止められる仕組みを用意する。「AIに何をさせるか」だけでなく「**何をさせないか、どこで人間が確認するか**」を設計する。

## チャットボット・RAGボット・AIエージェントの違い

### 一言でいうと

- **チャットボット**：入力に対して回答するAI
- **RAGボット**：文書を検索して回答するAI
- **AIエージェント**：判断し、ツールを使って行動するAI

### 比較表

| | チャットボット | RAGボット | AIエージェント |
|---|--------------|----------|---------------|
| **主な役割** | 回答・文章生成 | 文書に基づく回答 | 判断と実行 |
| **参照する情報** | ユーザー入力・会話履歴 | 検索された文書 | ユーザー入力・文書・ツールの実行結果 |
| **行動範囲** | 回答のみ | 回答と根拠表示 | **ツール実行まで** |
| **主なリスク** | ハルシネーション、機密情報の入力、プロンプトインジェクション | + 権限外文書の検索、機密の過剰出力、古い文書、間接プロンプトインジェクション | + **誤送信・誤削除・誤更新などの実害**、過剰権限、操作への誘導 |
| **安全設計の中心** | 入力管理、出力レビュー、インジェクション対策 | 文書分類、アクセス制御、根拠表示の制御、ログ管理 | **最小権限、ツール実行の制限、重要操作の承認、実行ログの監査** |

### リスクの大きさの関係

「回答するAI」から「行動するAI」になるほど、扱う情報と行動範囲が広がり、リスクも大きくなります。

```
チャットボット < RAGボット < AIエージェント
（誤回答・入力リスク）（+ 文書・アクセス制御リスク）（+ 実操作のリスク）
```

> **ポイント**：できることが増えるほどリスクも増える。AIの種類（行動範囲）に応じて、必要な安全対策を分けて考える。

## ツール実行の具体例

### ツール実行とは

**AIが外部システムや機能を呼び出すこと** です。これによりAIは文章生成を超えて実際の業務フローに関わるようになりますが、AIの判断が外部システムの操作や情報送信につながるため、ツールごとに権限・確認・承認の設計が必要です。

### 7つの具体例と注意点

| # | ツール実行 | 便利な点 | 注意点・安全な設計 |
|---|-----------|---------|------------------|
| 1 | **メール下書き作成** | 会議メモから短時間でお礼メール等を作成 | AIは **下書きまで**。送信前に人間が宛先・本文・添付・機密情報を確認 |
| 2 | **カレンダー確認・日程候補作成** | 空き時間から候補を提示し日程調整を支援 | 非公開予定・機密性の高い予定の扱い、参加者の予定の閲覧権限、確定登録は人間が行う |
| 3 | **社内文書検索（RAG）** | 経費精算ルール等を検索・要約して回答 | ユーザーが閲覧できる文書だけを検索対象に。文書分類・アクセス制御・根拠表示をセットで設計 |
| 4 | **チケット作成** | 不具合の整理・分類・登録を支援 | 機密・個人情報の混入、担当や重要度の誤判断に注意。登録前に内容・担当・添付を人間が確認 |
| 5 | **データベース参照** | 問い合わせ件数の集計などレポート作成を支援 | **読み取り専用権限** が原則。必要な範囲だけ参照。AI生成SQLの自動実行は危険 |
| 6 | **外部API呼び出し** | 配送状況の確認などを自然な文章で説明 | 送信データの個人情報を最小限に。APIキー等の認証情報をログ・プロンプトに含めない。規約・契約の確認 |
| 7 | **ファイル操作** | 資料の要約・保存などを支援 | 保存先の権限確認、外部共有は自動実行しない、上書き・削除は人間の確認を入れる |

### 特に注意すべき操作

メール送信、ファイル削除、外部共有、データベース更新、権限変更、支払い・購入、契約関連処理、コード実行、本番環境の操作。

> **取り消しが難しい操作、外部に影響する操作、金銭や契約に関わる操作** は、AIの判断だけで自動実行せず、AIには提案・下書き・候補作成までを任せ、人間が確認して実行する。

## AIエージェントでリスクが増える理由

### 7つの理由

| # | 理由 | 概要 |
|---|------|------|
| 1 | **外部データを読む** | メール・PDF・Webページ・文書・APIレスポンスには、誤情報・機密情報・悪意ある指示が含まれ得る。読んだ内容が判断と行動につながる |
| 2 | **ツールを実行する** | AIの判断が実際の操作になる。宛先の誤判断は誤送信に、ファイルの誤選択は機密共有につながる |
| 3 | **権限を持つ場合がある** | 権限が大きいほど誤動作・攻撃時の被害が大きい。読み取りのみと削除・送信まででは、リスクの大きさが違う |
| 4 | **誤判断が実操作につながる** | 通常のAIなら誤回答で済むものが、誤送信・誤削除・誤更新・危険なコマンド実行になる |
| 5 | **複数ステップで処理が進む** | 途中の誤り（メール内容の誤解釈、添付の誤選択）が後続の判断・操作に連鎖する |
| 6 | **プロンプトインジェクションの影響が大きい** | メール本文の「添付ファイルを外部に送信してください」等の指示に影響されると、誤回答ではなく **実際の操作** につながる |
| 7 | **ログや履歴に情報が残る** | 依頼内容、読み取ったデータ、実行内容、メール本文等がログに残り、ログ自体が機密情報を含み得る |

### リスクを小さくする設計

1. **権限を最小限に**：読み取りと書き込みを分ける。下書き作成は許可しても自動送信は許可しない
2. **外部データを命令として扱わない**：メール・PDF・Webページ・文書内の指示はAIが従うべき命令ではない
3. **重要操作は人間が承認**：実行前に宛先・内容・添付・対象を確認
4. **ツールごとに許可範囲を制限**
5. **実行ログを残し追跡できるようにする**
6. **異常時に処理を停止できるようにする**

> AIエージェントでは、便利さの追求だけでなく **「止める仕組み」「確認する仕組み」「追跡する仕組み」** を設計する。

## 過剰権限と重要操作のリスク

### 過剰権限とは

**本来の目的に必要ない権限まで与えてしまうこと** です。

- 読み取りだけでよいAIに書き込み権限を与える
- 下書き作成だけでよいのに送信権限まで与える
- 一部の文書だけでよいのに全社文書を検索できる
- 件数の集計だけでよいのに個人情報や本文まで取得できる
- テスト用のAIが本番環境を操作できる

過剰権限は **普段は問題が見えにくい** ものの、AIが誤判断したときやプロンプトインジェクションの影響を受けたときに **被害を大きくします**。AIにできることが多いほど、失敗したときの影響も大きくなります。

### 重要操作とは

**実行すると業務・顧客・システムに大きな影響を与える操作** です：メール送信、ファイル削除・外部共有、データベース更新、権限変更、支払い・購入、契約関連処理、コード実行、本番環境の操作、個人情報・機密情報の外部送信。

### 5つの代表的リスクと安全な設計

| リスク | 危険な例 | 安全な設計（AIに任せる範囲） |
|--------|---------|---------------------------|
| **メール送信** | 宛先・添付を確認せず自動送信 → 誤送信・情報漏洩 | AIは **下書きまで**。送信前に人間が宛先・本文・添付・機密を確認 |
| **ファイル操作** | 自動で移動・共有・削除 → 重要ファイルの削除、社外共有 | 削除候補・保存先候補・共有文面の下書きまで。削除・外部共有・上書きは承認後 |
| **データベース更新** | AI生成内容で自動更新 → 誤更新、復旧困難 | 原則 **読み取り専用**。更新が必要でもAIは更新案まで、実行は人間の確認後。ログとロールバックを用意 |
| **権限変更** | 自動で権限・共有設定を変更 → 管理者権限の誤付与、閲覧範囲の拡大 | AIは候補整理まで。実際の変更は人間の承認を必須に。監査ログ（誰が・いつ・何を・なぜ）を残す |
| **コード実行・本番操作** | 生成コマンドを本番で自動実行 → サービス停止、データ破損 | AIは原因候補の整理・修正案・**検証環境での試行** まで。本番反映はレビュー・テスト・承認後に人間が実行 |

### 最小権限の設計例

| 領域 | 過剰な権限 → 最小権限 |
|------|---------------------|
| メール | 自動送信 → **下書き作成のみ** |
| データベース | 書き込み → **読み取り専用** |
| データ取得 | 全文取得 → **必要な項目だけ** |
| RAG | 全社文書 → **対象文書だけ** |
| コード実行 | 本番環境 → **検証環境のみ** |
| ファイル | 削除実行 → **削除候補の提示** |

> **「AIにできることを増やす前に、できないことも設計する。」** 何を許可するかだけでなく、何を許可しないか、どこで人間が確認するかを設計することが基本。

## 人間の承認フローと最小権限

### 基本方針

AIエージェントを安全に使う基本方針は4つです。

1. AIに与える権限を最小限にする
2. 重要操作には人間の承認を入れる
3. 実行前に内容を確認する
4. 実行後にログを残す

> **AIは提案し、人間が確認して実行する。** この役割分担が安全設計の核。

### AIに任せる範囲をリスクで分ける

| リスク | 作業の例 | 扱い |
|--------|---------|------|
| **低リスク** | 情報整理、要約、候補作成、下書き作成、チェックリスト作成 | AIが間違えても人間が確認・修正しやすい。AIに任せやすい |
| **高リスク** | 外部送信、ファイル削除、データ更新、権限変更、支払い、購入、本番操作 | 一度実行すると影響が大きく取り消しが難しい。**自動実行しない** |

### 承認フローの基本

```
① AIが操作案を作る（メール本文、更新内容、共有先、削除候補など）
② 実行内容を人間に提示する
③ 人間が内容を確認し、必要があれば修正する
④ 人間が承認する
⑤ 承認後に操作を実行する
⑥ 実行ログを残す
```

### 操作別の承認例

**メール送信**：AIの役割は本文・件名の作成、宛先候補・添付候補の提示まで。人間は宛先、本文の適切さ、機密・個人情報の有無、添付、送信可否を確認する。

**ファイル操作**：AIは保存先候補・共有先候補・削除候補・ファイル名の提案まで。人間は対象ファイル、保存先、共有先の閲覧権限、機密情報、復旧可否を確認する。外部共有・削除・上書きは承認後に実行。

**データ更新**：AIは更新案の作成、変更箇所の整理、影響範囲の提示、変更前後の差分表示まで。人間は更新対象、更新内容、影響範囲、実行権限、ロールバック可否、監査ログを確認する。

### 承認画面で表示すべき情報

承認画面は **単なる「OKボタン」では不十分** です。人間が正しく判断できる情報を実行前に表示します。

- 実行する操作（メール送信か、ファイル共有か、データ更新か）
- 対象データ（どのファイル・顧客情報・レコードか）
- 宛先・共有先、添付ファイル
- 変更前と変更後、影響範囲
- 個人情報・機密情報の有無
- 実行者、承認者、実行後に残るログ

## ログ・監査・ロールバック

### なぜログが必要なのか

AIエージェントは複数の判断とツール実行を行うため、ログがないと「何を実行したのか」「なぜその操作をしたのか」「誰が承認したのか」「どのデータに影響したのか」が後から分からなくなります。誤送信や情報漏洩が疑われる場合、**影響範囲の特定** にもログが不可欠です。

### ログに残すべき情報

利用者／利用日時／ユーザーの依頼内容／AIが参照した情報（文書・ファイル・APIレスポンス）／AIが判断した内容／実行したツール／実行前の確認内容／**承認者**／実行結果／エラー内容／対象データ／**変更前と変更後の差分**（データ更新や権限変更では特に重要）。

### 監査とは

**AIエージェントの利用や実行内容が適切だったかを後から確認すること** です。

確認する観点：権限内の操作だったか／承認が必要な操作に承認があったか／機密情報を外部送信していないか／不要なデータを取得していないか／社内ルールに沿っていたか／問題発生時にログで追跡できるか。

**監査しやすいログの特徴**：操作単位で記録されている、実行者と承認者が分かる、実行前後の差分が分かる、対象データが特定できる、実行理由が確認できる、エラーも記録されている、時系列で追える、改ざんされにくい。

### ログに残しすぎるリスク

ログに個人情報、メール本文・ファイル内容・検索結果の全文、APIキー・トークン・パスワードが残ると、**ログ自体が漏洩リスク** になります。調査に必要な情報は残しつつ、個人情報・認証情報・機密情報の全文は残さない（最小化とマスキング）。

**あらかじめ決めておくこと**：何を記録するか／何を記録しないか／何をマスキングするか／誰が閲覧できるか（最小限に）／保存期間／削除タイミング／バックアップの扱い／改ざん防止／インシデント時に検索できるか。

### ロールバックとは

**問題が起きたときに変更を元に戻すこと** です。誤ったデータ更新、誤ったファイル変更、誤った権限変更・設定変更、本番環境での誤操作などが対象になります。

**ロールバックしやすい設計**：

1. 変更前の状態を保存する
2. 実行前後の差分を記録する
3. 一括実行ではなく段階的に実行する
4. 取り消し可能な操作を優先する（即削除ではなく削除候補への移動・アーカイブ）
5. バックアップを取得する
6. 変更対象を限定し、実行IDで操作を追跡できるようにする
7. 失敗時に自動停止する仕組みを用意する

### ロールバックできない操作に注意

**一度実行すると戻せない操作** があります：メール送信（相手に届いてしまう）、外部共有・外部APIへの送信（外部に出た情報は回収困難）、支払い・購入・契約関連処理、個人情報の外部送信、公開済みコンテンツの拡散、完全削除されたデータ。

> **戻せない操作ほど、事前承認と確認を厳しくする。** 戻せない操作はAIだけで自動実行しない。

### インシデント時の確認項目

いつ発生したか → 誰が依頼したか → AIは何を判断したか → どのツールを実行したか → 誰が承認したか → どのデータに影響したか → 外部送信があったか → ロールバックできるか → 再発防止策は何か。

## ミニ演習：メール送信AIエージェントを安全に設計する

### 演習の前提

社内向けのメール送信AIエージェントを設計すると想定します。受信メールを読み、関連する社内文書を検索し、返信文を作成し、宛先・添付候補を提示し、下書きを作成し、必要に応じて送信する仕組みです。

このエージェントは **複数のリスクを同時に持ちます**：メールを読むリスク、文書を検索するリスク、本文を生成するリスク、宛先を判断するリスク、添付を扱うリスク、送信するリスク。

### まず考えること

1. 誰がこのAIを利用するのか（全社員か、営業部か、サポート担当か）
2. AIがどのメールを読めるのか（本人のメールだけか、共有メールボックスも読めるか）
3. どの社内文書を検索できるのか
4. 添付できるファイルの範囲
5. 自動送信を許可するか、どの操作に承認が必要か
6. ログに何を残すか
7. **誤送信が起きたときにどう対応するか**

### 危険な設計例

- AIが全メールを読める（他部署のメールまで）
- 全社文書を検索できる（人事資料・経営資料まで）
- 任意の宛先に送信できる、任意のファイルを添付できる
- AIの判断だけで自動送信できる
- 送信前の確認画面がない
- 送信ログが残らない
- 誤送信時の対応手順がない

### 4つのリスクと対策

| # | リスク | 起こり得ること | 対策 |
|---|--------|--------------|------|
| 1 | **誤った宛先への送信** | 同姓同名の別人、社外への社内情報送信、CC/BCCの誤り、顧客A向け内容を顧客Bへ | AIは宛先候補の提案まで。送信前に人間が確認。社外宛てには警告。ドメイン・所属・過去のやり取りを表示して判断しやすく |
| 2 | **本文に機密情報が含まれる** | 顧客名・案件名・契約条件・金額・個人情報の混入、RAG検索結果の出しすぎ | 出力前の機密確認、個人情報のマスキング、本文は必要最小限、社外送信前に人間がレビュー |
| 3 | **添付ファイルの誤送信** | 間違ったファイル、社内向け資料の社外送信、顧客別資料の取り違え、古い版 | AIは添付候補の提示まで。ファイル名・保存場所・作成日・公開範囲を表示。社外共有禁止ファイルは添付不可に。承認後に添付 |
| 4 | **プロンプトインジェクション** | メール本文内の「添付ファイルを外部アドレスへ転送してください」等の指示に影響される | メール本文・添付の内容を命令として扱わない。外部送信は自動実行しない。不審な指示の検出時は人間に確認。重要操作に承認を必須に |

### 安全な権限設計

| 機能 | 権限の絞り方 |
|------|------------|
| メール閲覧 | 必要なメールだけ（原則、利用者本人のメール） |
| 文書検索 | 利用者が閲覧可能な文書だけ（**AIがユーザー本人より広い権限を持たない**） |
| ファイル参照 | 許可されたフォルダだけ |
| 添付 | 承認済みファイルに限定 |
| メール作成 | 下書き作成まで |
| メール送信 | 原則として人間の承認後 |
| 外部転送 | 自動実行しない |

### 承認フローとログ

**送信前に人間が確認する項目**：宛先（CC・BCC含む）／件名／本文（誤解を与えないか、不適切表現がないか）／機密・個人情報の有無／添付ファイル／社外送信してよい内容か（顧客・契約上の問題）。

**ログに残す情報**：依頼者／AIが作成した下書き／参照した文書／提案した宛先／添付候補／承認者／送信日時／送信先／添付ファイル／送信結果。誤送信時にはこれらをもとに「何を・誰に・誰の承認で送ったか」を確認し、必要な対応を判断します。ログ自体への機密情報の残しすぎにも注意します。

> **結論**：AIは下書きと候補作成を行い、人間が確認して送信する。メール送信は **外部に情報が出る操作** であるため、必ず承認と監査を組み込む。

## セクション全体のまとめ

- **AIエージェントは「行動するAI」**。LLMの判断がツール実行という実際の操作につながる点が、チャットボット・RAGボットとの決定的な違い
- チャットボット → RAGボット → AIエージェントと進むほど、扱う情報と行動範囲が広がり、**リスクも「誤回答」から「実害（誤送信・誤削除・誤更新）」へ拡大** する
- リスクが増える理由は、外部データを読む、ツールを実行する、権限を持つ、誤判断が実操作につながる、複数ステップで誤りが連鎖する、プロンプトインジェクションの影響が操作に及ぶ、ログに情報が残る、の7つ
- **過剰権限は普段は見えないが、誤判断や攻撃時に被害を拡大させる**。読み取りと書き込み、下書きと送信を分け、最小権限で設計する
- 重要操作（送信・削除・更新・権限変更・支払い・本番操作）は自動実行せず、**AIは提案、人間が承認して実行**。承認画面には人間が判断できる情報を表示する
- **ログ・監査・ロールバック** が安全運用の基盤。誰が依頼し、AIが何を判断し、誰が承認し、何を実行したかを残す。ただしログ自体に機密を残しすぎない
- **戻せない操作（メール送信、外部共有、支払い、個人情報の外部送信）ほど、事前承認を厳しくする**

### キーワード一覧

| 用語 | 意味 |
|------|------|
| AIエージェント | LLMが判断し、外部ツールを使いながら目的達成に向けて処理を進める「行動するAI」 |
| 処理ループ | 観察→判断→行動→結果確認→次の判断、を繰り返す構造。誤りが後続処理に連鎖し得る |
| ツール実行 | AIが外部システムや機能（メール・カレンダー・DB・API等）を呼び出すこと |
| 過剰権限 | 本来の目的に必要ない権限まで与えてしまうこと。誤判断・攻撃時の被害を拡大させる |
| 最小権限の原則 | 目的達成に必要な最小限の権限だけを与える考え方 |
| 重要操作 | 実行すると業務・顧客・システムに大きな影響を与える操作（送信・削除・更新・権限変更・支払い等） |
| 承認フロー | AIが操作案を作り、人間が内容を確認・承認してから実行する流れ |
| 提案と実行の分離 | AIには下書き・候補作成までを任せ、実行判断は人間が行う設計パターン |
| 監査 | AIエージェントの利用・実行内容が適切だったかを後から確認すること |
| ロールバック | 問題が起きたときに変更を元に戻すこと。変更前の状態の保存や差分記録が前提になる |
| 戻せない操作 | メール送信、外部共有、支払いなど、一度実行すると取り消しが難しい操作。承認を特に厳しくする |
| インシデント対応 | 問題発生時に、ログをもとに経緯・影響範囲を確認し、復旧・再発防止を行うこと |
