## AIの出力はなぜリスクになるのか

### 基本の考え方

**AIの出力は最終成果物ではなく、確認が必要な下書き** です。自然で説得力のある文章でも、必ずしも正しい・安全とは限りません。

### AI出力の主なリスク

| リスク | 内容 |
|--------|------|
| **誤情報** | もっともらしい文章で間違った情報を出す。根拠不明な断定、古い情報をもとにした回答 |
| **機密情報・個人情報の混入** | 会話履歴やRAGで参照した文書の内容が回答に混ざる。システムプロンプトが表示されることも |
| **不適切な表現** | 差別的・攻撃的・失礼な表現、「必ず」「絶対に」といった断定しすぎる表現、ブランドに合わない表現、法的責任を認めるような表現 |
| **危険な手順・助言** | セキュリティ上危険な設定の提案、データ削除につながる操作の安易な案内、医療・法律・金融での誤ったアドバイス |
| **脆弱なコード** | 入力検証の不足、認証・認可の不備、古いライブラリの使用、秘密情報の埋め込み |
| **後続処理の誤動作** | SQL・コマンド・HTML・JSONをそのまま実行・表示・処理することによる問題 |

> **重要な区別**：文章が自然であること ≠ 正しいこと ≠ 安全であること。「役立つように見える回答」が安全とは限らない。

### 用途に応じてリスクを分ける

| 用途 | 例 | 扱い |
|------|-----|------|
| **低リスク** | アイデア出し、下書き、文章のたたき台、学習補助 | 比較的気軽に使える（確認は必要） |
| **高リスク** | 顧客への送信、契約・法務判断、医療・金融・人事に関する判断、コードの本番利用、外部公開、自動実行 | **人間による確認を必須にする** |

> **安全な使い方の基本**：AIには下書きや候補を作らせ、人間が最終判断を行う。

### AI出力を使う前の確認ポイント

1. 事実が正しいか、根拠を確認できるか、最新か
2. 機密情報・個人情報が含まれていないか
3. 表現が場面・相手・ブランドに合っているか
4. 法務・契約上問題のある内容がないか
5. 出力形式（JSON、表、メール文、コード）が正しいか
6. 人間の確認が必要な内容ではないか

## ハルシネーションとは何か

### 定義

**ハルシネーションとは、AIが事実ではない内容を、もっともらしく生成してしまう現象** です。

典型例：「この制度の申請期限は？」→ AIが自信を持って「毎月15日です」と回答 → 実際の規程では毎月10日。**あいまいではなく断定的な文章で出力される** ため、ユーザーは正しい情報のように受け取ってしまいます。

### ハルシネーションで起こること

- 存在しない制度・手当を説明する
- 存在しない法律・規程・論文・資料を引用する
- 誤った数値・日付を出す
- 古い情報を最新情報のように答える
- 根拠がない内容を断定する、事実と推測を混ぜる

### なぜ起こるのか

LLMは **入力された文脈をもとに自然な文章を生成するモデル** であり、常に事実確認を行ってから回答しているわけではありません。

- 知らないことでも、文脈からそれらしい回答を生成してしまう
- 根拠が不十分でも、文章として自然になるように内容を補完する
- **質問の前提が間違っていても、その前提を受け入れて回答する**ことがある（「申請期限は15日ですよね」→ 違っていても前提に沿って回答）

### RAGでもハルシネーションは起こる

RAGは根拠に基づいた回答をしやすくする有効な対策ですが、**完全にはなくなりません**。

- 関係ない文書・古い文書を根拠にしてしまう
- 検索結果をAIが誤って解釈する
- 文書に書かれていない内容を補完してしまう
- 複数の文書の内容を混同する
- 根拠文書と違う回答をしてしまう

### ハルシネーションを見抜くポイント

1. 根拠が示されているか。出典は **実在するか**
2. 数値や日付が出ていないか（少しの違いでも影響が大きい）
3. 最新情報が必要な内容ではないか（制度・料金・仕様・法律は変わる）
4. 断定しすぎていないか
5. 質問の前提が正しいか
6. 文書に書かれていない内容をAIが補っていないか

→ 重要な情報は、**公式資料や一次情報** で確認する。

### ハルシネーションを減らす工夫

- 根拠文書を与える（何も与えずに答えさせない）
- 回答に根拠を表示させる
- 不明な場合は「わかりません」と答えさせる
- 推測と事実を分けて書かせる
- 高リスクな回答は人間がレビューする
- RAGの文書を更新し、古い文書が検索されないよう管理する

### AIに任せすぎてはいけない領域

法律・医療・金融に関する判断、人事評価や契約判断、セキュリティ設定や本番環境の操作、顧客への正式回答、外部公開資料、経営判断に使う資料。

> AIは情報整理や下書きには有効だが、**最終的な判断や責任をAIに任せない**。AIの回答は「正解」ではなく「確認が必要な出力」。

## 業務でハルシネーションが問題になるケース

### 8つの代表的なケース

| # | 場面 | 誤りの例 | 起こり得る影響 |
|---|------|---------|--------------|
| 1 | **顧客対応** | 存在しない機能の案内、料金プランの間違い、サポート対象外を対象内と回答、古いFAQに基づく回答 | クレーム、信用低下、契約トラブル、サポート負荷増 |
| 2 | **営業・提案資料** | 製品仕様の誤記載、未対応機能を対応済みと記載、導入実績・価格の誤り、不正確な競合比較 | 顧客への誤説明、契約後の「聞いていた話と違う」 |
| 3 | **社内規程・人事制度** | 休暇制度の条件・申請期限の誤案内、経費精算ルールの誤説明、古い規程に基づく回答 | 社員の誤った手続き、不公平感やトラブル |
| 4 | **契約・法務** | 契約解除条件の誤要約、責任範囲の誤説明、存在しない条文・判例の引用、不利な条件の見落とし | 契約リスク、取引先トラブル、損害・責任の発生 |
| 5 | **開発・技術対応** | 存在しないAPIを使うコード、古い仕様の前提、脆弱なコード、誤ったエラー原因の説明、危険な操作の提案 | システム障害、脆弱性、データ損失 |
| 6 | **医療・金融・専門領域** | 医療情報の断定的な説明、金融リスクの過小評価、税務・会計の誤案内、法規制の誤解釈 | 利用者への大きな不利益 |
| 7 | **意思決定資料** | 市場規模の誤記載、存在しない調査結果の引用、古いデータを最新のように扱う、根拠のない断定的分析 | 経営判断・投資判断の誤り |
| 8 | **広報・外部公開** | 事実と異なる発表、未公開情報の混入、誤解を招く表現、公式見解のような断定 | 拡散後の回収困難、メディア対応 |

### 問題が大きくなりやすい条件

- AI出力を **そのまま**（人間の確認なしに）使う場合
- 根拠を確認しない場合
- 最新情報が必要な内容（料金・規程・法律・仕様）
- 数値や日付が重要な場合
- 顧客や社外に共有する内容
- 法務・医療・金融など専門性が高い領域
- **自動送信や自動実行につながる場合**（誤りがそのまま実害になる）
- 責任ある判断に使う場合

### 業務で使う前の確認ポイント

「文章として自然か」ではなく、「**事実として正しいか**」「**業務で使ってよいか**」を確認します。

1. 事実が正しいか、根拠を確認できるか、出典は実在するか
2. 最新情報か。数値・日付は正しいか
3. 社内規程や業務ルールと一致しているか
4. 専門家や担当部署の確認が必要な内容ではないか
5. 顧客や社外に出してよい内容か。機密情報・未公開情報が含まれていないか

## AIの出力をそのまま公開してはいけない理由

### 「公開」の範囲は広い

Webサイト掲載だけでなく、SNS投稿、メルマガ配信、プレスリリース、顧客向けメール、提案書、社外向け資料、FAQ・ヘルプページ、アプリ内でユーザーに表示する文章——**社外の人が見る可能性があるものはすべて公開物** として扱います。

### 7つの理由

| # | 理由 | ポイント |
|---|------|---------|
| 1 | **誤情報が含まれる可能性** | 存在しない機能、古い料金・仕様、誤った実績・数値・日付、存在しない出典。公開されれば顧客や取引先への誤解、クレーム、契約トラブルに |
| 2 | **機密情報が混ざる可能性** | 社内メモ、未公開の事業計画、開発中の製品情報、社内限定の価格情報。RAGの参照文書や会話履歴の内容が混ざることも |
| 3 | **個人情報が含まれる可能性** | 氏名・連絡先・所属だけでなく、**文脈から個人が推測される** 情報にも注意（名前を消しても安全とは限らない） |
| 4 | **不適切な表現が含まれる可能性** | 「必ず成功します」「絶対に安全です」等の断定、失礼な言い回し、ブランドに合わない表現、法的責任を認める表現 |
| 5 | **著作権・権利の問題** | 他社の文章に似た表現、引用元不明の説明、商標の使い方、第三者資料の公開利用、ライセンス条件が不明なコード。**AIが生成したからといって自由に使えるとは限らない** |
| 6 | **会社の公式見解に見える** | 社外に出た文章は会社の発信として見られる。未確定の計画を確定事項のように書く、顧客への約束に見える表現は後から問題になる |
| 7 | **拡散後の修正が難しい** | SNS拡散、スクリーンショット、検索エンジンのキャッシュ、メール転送。**一度広がった情報の完全な回収は困難** |

> 公開前のチェックは、公開後の謝罪や訂正対応よりもずっと重要。

### 安全な公開フロー

```
① AIで下書きを作る
② 人間が内容を確認（事実・数値・日付／機密情報・個人情報）
③ 表現とトーンを調整（公開先・ブランド・誤解の有無）
④ 必要に応じて法務・広報・担当部署が確認
⑤ 公開
```

> **ポイント**：AIには下書きを作らせ、**公開判断は人間が行う**。公開するかどうかを判断し、責任を持つのは人間。

## Insecure Output HandlingとAI生成コードのリスク

### Insecure Output Handlingとは

**AIの出力を十分に検証せず、そのまま後続処理に使ってしまう問題** です。AIの出力が画面表示にとどまらず、**実行・保存・送信・更新** につながる場合、リスクは大きくなります。

### なぜ危険なのか

- ユーザーの不正な指示や外部文書の不審な指示（プロンプトインジェクション）の影響が、出力に混ざる可能性がある
- AIは必ずしも決められた形式で出力するとは限らない（JSONに余計な文章が混ざる、想定外の値が含まれる）

→ **AIの出力は信頼済みのデータではなく、外部入力と同じように検証する。**

### 4つの危険なパターン

| パターン | 何が起こるか | 対策 |
|---------|------------|------|
| **HTMLをそのまま表示** | 不正なスクリプトが含まれているとXSS（クロスサイトスクリプティング）に。セッション情報の窃取、不正な画面表示 | 表示前にサニタイズ、許可するタグ・属性を限定、プレーンテキストとして表示 |
| **SQLをそのまま実行** | 意図しないデータ取得・更新・削除、権限外データの参照 | 直接実行しない。読み取り専用に限定、許可テーブルの制限、人間の確認、クエリの静的検証 |
| **コマンドをそのまま実行** | ファイル削除、設定破壊、情報漏洩、本番環境の停止 | 意味を理解してから実行。削除・上書き・権限変更・本番変更は特に慎重に。安全な環境で検証 |
| **JSONをそのまま信頼** | 想定外の宛先への送信、機密ファイルの添付フラグ、想定外のアクション名による誤動作 | **スキーマ検証** を必ず行い、実行できるアクションや値は **許可リスト** で制御 |

### AI生成コードのリスク

「動くコード」が「安全なコード」とは限りません。

- 入力検証の不足（ユーザー入力をそのまま処理）
- 認証・認可の不備（未ログインでも操作できる、一般ユーザーが管理者機能を使える）
- エラーハンドリングが弱い（エラーに内部情報が出る）
- 秘密情報をコードに埋め込む例を生成する
- 古いライブラリ・非推奨APIの使用、ライセンス上の問題、テスト不足

**特に注意すべきコードの例**：

- SQLを文字列結合で組み立てる → SQLインジェクションのリスク
- ユーザー入力をそのままHTMLに表示 → XSSのリスク
- 認証チェックの省略、管理者権限を前提にした処理
- APIキーの直書き、HTTPSではなくHTTPの使用
- エラー内容をそのまま画面に表示、ファイルパスを検証せずに使用

### AI生成コードの確認ポイント

1. 入力検証があるか
2. 認証と認可が適切か（誰が使えるのか、権限チェックがあるか）
3. エラー処理が安全か（内部情報・秘密情報が含まれないか)
4. コード内にAPIキーやパスワードがないか
5. 依存ライブラリが安全か、古くないか
6. ライセンス上の問題がないか
7. テストが十分か、本番環境で実行してよい処理か

> AI生成コードは **レビュー、テスト、静的解析、セキュリティ確認** を通してから利用する。

## ミニ演習：AI出力レビュー

### レビューの基本観点

事実は正しいか／根拠を確認できるか／数値・日付は正しいか／最新か／機密情報・個人情報が含まれていないか／表現は適切か／出力形式は正しいか／そのまま公開・送信・実行してよいか。

### 判断は3段階で考える

| 段階 | 内容 | 例 |
|------|------|-----|
| **① そのまま使える** | 低リスク用途で内容確認も済んでいる | 社内メモ用のアイデア出し、機密を含まないたたき台 |
| **② 修正すれば使える** | 表現の修正や根拠確認が必要な部分がある。**AI出力の多くはここに該当** | 断定表現を含む製品紹介文 |
| **③ そのまま使ってはいけない** | 重大な誤情報、機密情報、個人情報、危険な手順、脆弱なコードを含む | 顧客情報入りのメール、SQLインジェクションのあるコード |

> 迷った場合は、**公開しない・送信しない・実行しない**。

### ケーススタディ

#### ケース1：顧客向けメール → そのまま使ってはいけない

> A社の田中様。先日の商談でご相談いただいた件について、御社の来期予算1,200万円に合わせたプランをご提案します。競合B社よりも安価に提供可能です。

顧客名・担当者名・来期予算・競合名が含まれ、「競合B社よりも安価に提供可能」という根拠不十分な断定表現もある。マスキング・表現修正のうえ、**人間が確認して送信** する。

#### ケース2：製品紹介文 → 修正すれば使える

> この製品は、すべての業務システムと自動連携できます。導入すれば、作業時間を必ず50%削減できます。

「すべての業務システム」は過剰、「必ず50%削減」は保証と受け取られかねない。**対応範囲を正確に書き、数値には根拠を示し、断定表現を避ける** 修正が必要。

#### ケース3：社内FAQ回答 → 根拠確認が必要

> 有給休暇の申請期限は、取得予定日の前日までです。ただし、急な体調不良の場合は事後申請が可能です。

一見自然だが、申請期限・事後申請の条件・例外条件が **最新の公式文書と一致しているか** の確認が必要。根拠文書を表示できるかも重要。

#### ケース4：外部公開記事 → そのまま使ってはいけない

> 当社は来年度、新しいAIサービスを正式リリースします。すでに大手企業数十社への導入が決定しています。

未公開情報の可能性、将来計画の確定事項のような記載、「数十社への導入決定」の根拠確認が必要。会社の公式発表として受け取られるため、**広報・法務・事業担当部門の確認** が必要。

#### ケース5：AI生成コード（SQL文字列結合） → そのまま使ってはいけない

ユーザー入力をそのままSQL文に埋め込むコードは **SQLインジェクションのリスク**。プレースホルダの使用、入力検証、最小権限、テストの有無を確認する。

#### ケース6：AI生成コマンド（ログ削除） → 慎重な確認が必要

削除対象は正しいか、本番環境ではないか、バックアップはあるか、コマンドの意味を理解しているか、実行前に承認が必要ではないか。**意味を理解せずにコピーして実行しない**。

#### ケース7：AI生成JSON → そのまま処理してはいけない

> { "action": "send_email", "to": "customer@example.com", "attach_internal_document": true }

外部送信 + 内部文書の添付指定。そのまま処理すると **機密文書を外部送信** してしまう可能性がある。スキーマ検証、許可された値のみ受け付け、重要操作は人間の承認後に実行する設計にする。

### AI出力レビューシートの観点

1. そのまま使えるか（3段階判断）
2. 誤情報がないか、根拠を確認できるか
3. 数値・日付が正しいか
4. 機密情報・個人情報が含まれていないか
5. 表現が相手・用途に適切か
6. 権利・契約上の問題がないか
7. 実行・送信・公開してよいか
8. 人間の承認が必要か

## セクション全体のまとめ

- **AIの出力は完成品ではなくレビュー対象**。自然な文章であることと、正しいこと・安全であることは別
- **ハルシネーション** はAIが事実ではない内容をもっともらしく生成する現象。LLMは事実確認をしてから回答しているわけではなく、RAGを使っても完全にはなくならない
- 業務では、顧客対応、営業資料、社内規程、契約・法務、開発、専門領域、意思決定資料、広報など幅広い場面でハルシネーションが問題になる。**自動送信・自動実行につながる場合は誤りがそのまま実害になる**
- 出力の公開前には、誤情報・機密情報・個人情報・不適切表現・権利問題・公式見解性を確認する。**一度拡散した情報の回収は難しい** ため、公開前チェックを重視する
- **Insecure Output Handling** はAI出力を検証せず後続処理に使う問題。HTML・SQL・コマンド・JSON・コードは外部入力と同じように検証する。「動くコード」≠「安全なコード」
- レビューは「そのまま使える／修正すれば使える／使ってはいけない」の3段階で判断し、迷ったら公開・送信・実行しない。**AIには下書きや候補を作らせ、人間が確認して最終判断する**

### キーワード一覧

| 用語 | 意味 |
|------|------|
| ハルシネーション | AIが事実ではない内容をもっともらしく生成してしまう現象 |
| 根拠確認 | 回答の出典が実在し、内容と一致しているかを確認すること |
| 前提の誤り | 質問に含まれる誤った前提を、AIがそのまま受け入れて回答してしまうこと |
| 高リスク用途 | 顧客送信、契約・法務判断、医療・金融・人事、本番利用、外部公開、自動実行など、人間の確認を必須にすべき用途 |
| Insecure Output Handling | AIの出力を十分に検証せず、そのまま後続処理に使ってしまう問題 |
| XSS（クロスサイトスクリプティング） | 不正なスクリプトがWebページで実行される攻撃。AI生成HTMLの無検証表示で起こり得る |
| SQLインジェクション | 入力によってSQL文の意味が変わってしまう攻撃。文字列結合によるSQL組み立てで起こり得る |
| サニタイズ | HTMLなどから危険な要素を取り除き、安全な形にする処理 |
| スキーマ検証 | JSONなどの出力が想定した形式・値になっているかを検証すること |
| 許可リスト | 実行できるアクションや値をあらかじめ許可されたものだけに制限する方式 |
| 公開前チェック | 事実・表現・機密情報・個人情報・権利・責任範囲を、公開の前に人間が確認すること |
