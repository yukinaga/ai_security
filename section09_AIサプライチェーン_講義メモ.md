## AIサプライチェーンとは何か

### 定義

**AIサプライチェーンとは、AIを支えるデータ、モデル、ツール、サービス、運用のつながり** のことです。AIは単体で完結して動いているわけではなく、学習データ、事前学習済みモデル、AI API、ライブラリ、クラウド環境、RAG文書、ベクトルDB、外部ツールなど、多くの外部要素に依存しています。

**そのどこかに問題があると、AIシステム全体の安全性に影響します。** だからこそ、AIそのものだけでなく、AIを支える周辺要素まで含めて安全性を考える必要があります。

### 主な構成要素

データ提供元（学習データ・評価データ・RAG文書）／データ加工プロセス／モデル提供元・モデル配布サイト／AI API事業者／開発ライブラリ／文書管理基盤・ベクトルDB（RAG）／外部ツール（AIエージェント）／クラウド・インフラ／監視・ログ基盤。

### 構成要素ごとのリスク

| 要素 | 主なリスク |
|------|-----------|
| **データ** | 不正確・偏りのあるデータ、個人情報の混入、利用許諾が不明、攻撃者による混入、古いデータの使い続け |
| **モデル** | 出所不明、改ざん、悪意ある振る舞い、ライセンス未確認、更新による挙動変化、提供元のサービス停止 |
| **ライブラリ・ツール** | 脆弱なライブラリ、依存関係内の脆弱性、悪意あるパッケージ、過剰な権限、開発環境に残る秘密情報 |
| **外部AIサービス** | データの外部送信、保存・学習利用の扱い、契約・社内ルールとの不整合、サービス停止 |

RAGやAIエージェントを使うと、文書提供元、ベクトルDB、検索インデックス、外部ツール、実行ログなどが加わり、**サプライチェーンはさらに広がります**。

### サプライチェーン攻撃の考え方

**直接システムを攻撃するのではなく、依存先や取り込み元を通じて影響を与える攻撃** です。

- 学習データやRAG文書への悪意あるデータの混入
- 改ざんされたモデルの配布
- 悪意あるライブラリを使わせる（秘密情報の窃取）
- RAGへの不審な文書の取り込み（間接プロンプトインジェクション）
- 外部APIの応答を悪用したAIの判断誘導
- 開発環境からのAPIキー・トークンの窃取

### 管理の基本

1. **何に依存しているかを把握する**（データセット、モデル、ライブラリ、外部API、クラウドを一覧化）
2. 提供元・出所は信頼できるか、ライセンスや契約は問題ないか
3. 既知の脆弱性はないか、機密情報を送っていないか
4. 更新時に挙動が変わらないか
5. **問題発生時に停止や切り替えができるか**

> AIサプライチェーン管理は、技術的な確認だけでなく、**契約、権限、運用、監視** も含めて行う。

## 外部モデル・API利用のリスク

### 前提

外部モデル・AI APIを使うと、自社で大規模モデルを開発・運用しなくても高度なAI機能を組み込めます。一方で、**自社システムの一部を外部サービスに依存する** ことになるため、「単なる部品」として気軽に使うのではなく、**自社システムの一部として管理する** ことが重要です。

### 8つのリスク

| # | リスク | 概要・確認ポイント |
|---|--------|------------------|
| 1 | **データが外部に送信される** | ユーザー入力だけでなく、会話履歴・システムプロンプト・ファイル内容・RAG検索文書・メタデータも送信され得る。何が送信されるかを把握する |
| 2 | **保存・学習利用・削除の扱い** | 入力・出力データは保存されるか、保存期間は、モデル学習に使われるか、人間が内容を確認する可能性は、削除依頼はできるか |
| 3 | **個人情報・機密情報の送信** | 個人情報・顧客情報・社内文書・ソースコードに注意。**認証情報（APIキー・パスワード等）は絶対に送らない**。マスキング・匿名化・データ最小化を行う |
| 4 | **モデルの挙動が変わる** | 提供元の更新で同じ入力でも出力が変わる。出力形式の変化で後続処理が壊れることも。テスト・評価・監視で変化に気づける仕組みを |
| 5 | **サービス停止・障害・制限** | API障害、レート制限、モデル提供終了、仕様変更。**障害時の代替手段**（縮退運用、別モデルへの切り替え、ユーザーへの案内）を決めておく |
| 6 | **契約・規約・ライセンス** | 商用利用の可否、入力データの扱い、出力結果の利用条件、顧客情報の送信可否、業界規制。「技術的に使えるか」だけでなく「**契約上使ってよいか**」を確認 |
| 7 | **出力品質と責任範囲** | ハルシネーション、不適切表現、古い情報、脆弱なコード。**外部モデルを使っても、出力の最終責任は利用側に残る** |
| 8 | **ベンダーロックイン** | 特定サービスの仕様・独自機能・プロンプト最適化に強く依存すると移行が困難に。API呼び出しの抽象化、複数モデルでの評価などで移行可能性を確保 |

### 利用前の確認項目

どのデータを送信するか／個人情報・機密情報は含まれないか／保存・学習利用・削除条件／契約・規約・ライセンス／モデル更新時の影響をテストできるか／障害時の代替手段／出力を人間が確認する運用か／ログに何が残るか／コストと利用制限。

> **技術、契約、データ、運用、コストをまとめて確認** してから使う。

## オープンソースモデル利用の注意点

### メリットと前提

オープンソースモデル（重み・コード・設定等が公開されたモデル）は、自社環境での運用、外部へのデータ送信の削減、挙動の検証しやすさ、用途に合わせた調整（ファインチューニング）、ベンダー依存の低減などのメリットがあります。

ただし、**自由度が高い分、運用責任も利用者側に大きくなります**。「無料で使えること」と「安全に使えること」は別です。

### 7つの注意点

| # | 注意点 | 確認すること |
|---|--------|------------|
| 1 | **ライセンス確認** | 「オープン」と付いていても商用利用・改変・再配布・組み込みに条件がある場合がある。出力結果の利用条件、利用禁止用途、表示義務も確認 |
| 2 | **モデルの出所** | 公式リポジトリ・公式配布ページからの取得か。第三者の再アップロード・改変版ではないか。モデルカードの有無、更新履歴、ハッシュ・署名による正当性確認 |
| 3 | **モデルの安全性** | 外部APIと違い、安全制御は自分たちで考える必要がある。不適切な出力、ハルシネーションの多さ、プロンプトインジェクションへの耐性を自社用途で評価 |
| 4 | **学習データの不透明性** | 学習データの詳細が分からない場合、著作権・個人情報・偏り・古い情報のリスクを判断しにくい。出力確認・用途制限・人間のレビューを組み合わせる |
| 5 | **ファインチューニングのリスク** | 学習データに個人情報・機密情報・**認証情報** を含めない。契約上の確認、データ出所の明確化、学習後の情報漏洩の考慮、評価用データの分離 |
| 6 | **運用・保守の責任** | 推論環境のセキュリティ、モデルファイルのアクセス管理、ログ管理、脆弱性対応、バージョン管理、障害対応、コスト管理——すべて利用者側の責任 |
| 7 | **更新・派生モデルの管理** | 同じ名前に見えても異なるバージョン・改変版があり得る。バージョン固定、更新前の評価、派生モデルのライセンス確認、切り戻しの準備 |

### 利用前チェックリスト

- [ ] ライセンスに問題がないか（商用利用・組み込みの可否）
- [ ] 配布元が信頼できるか（公式リポジトリか）
- [ ] モデルカードや説明資料を確認したか
- [ ] 自社用途で安全性評価を行ったか
- [ ] 出力レビューの運用があるか
- [ ] 学習データや出力に関するリスクを理解しているか
- [ ] 運用環境のアクセス制御があるか
- [ ] バージョン管理と更新管理ができているか
- [ ] 問題発生時に停止や切り替えができるか

> **公開されているモデルをそのまま信頼しない。** 利用条件、安全性、運用責任を確認したうえで、目的に合った形で使う。

## Training Data Poisoningとは何か

### 定義

**Training Data Poisoningとは、AIの学習データに不正なデータを混入させ、モデルの出力や判断に悪影響を与える攻撃・リスク** です。

AIは学習データからパターンや判断の傾向を学ぶため、**学習データに問題があると、その問題がモデルの中に取り込まれてしまいます**。「本来は危険と判断すべきものを安全と判断する」「特定の条件でだけ意図しない振る舞いをする」といった問題につながります。

### 何が「毒」になるのか

誤ったラベルが付いたデータ／攻撃者が意図的に混入させたデータ／偏りの強いデータ・不正確なデータ／悪意ある指示を含むデータ／出所不明・改ざんされたデータ／古い情報を含むデータ／個人情報・機密情報を含むデータ（出力を通じた漏洩リスクにも）。

### 具体例

**分類モデルの例**：迷惑メール分類モデルの学習データに、本来は迷惑メールであるデータが「通常メール」として多数登録されると、モデルが迷惑メールの特徴を誤って学習し、**ブロックすべきメールを通してしまう** 可能性があります。

**チャットAIの例**：学習・調整データに誤った知識、不適切な表現、危険な手順の回答例が混ざると、モデルが似た出力をしやすくなります。社内向けモデルが社内ルールと違う回答を学習すると、業務での誤案内につながります。

### RAG文書の混入との違い

| | Training Data Poisoning | RAG文書の混入 |
|---|------------------------|--------------|
| 影響を受ける段階 | **モデルの学習段階** | 検索・回答の段階 |
| 影響の現れ方 | モデルの振る舞いそのものが変わる | 検索結果を通じて回答に影響する |
| 共通点 | **AIが参照するデータの信頼性が重要**（出所・品質・更新状況の管理） | 同左 |

### なぜ気づきにくいのか

- 学習データが大量で、すべてを人間が確認するのは現実的でない
- 影響が **一部の条件でしか出ない** ことがある（通常のテストでは検出しにくい）
- 出力の誤りがモデル由来かデータ由来か分かりにくい
- データの出所や変更履歴が追跡できないと原因調査が難しい

### リスクが高くなる場面

外部データセットの利用／ユーザー投稿データの学習利用（攻撃者が意図的に投稿できる）／Webから自動収集したデータ／出所不明のデータ／ラベル付けの外部委託／自社データでのファインチューニング／継続学習。

### 2つの対策

**対策1：データの出所と品質を管理する（学習前）**

- 提供元は信頼できるか、利用許諾・契約に問題はないか
- ラベルは正しいか、異常なデータ（特定パターンの不自然な増加等）が混ざっていないか
- 個人情報・機密情報を含まないか、古いデータ・重複データはないか
- **変更履歴を追跡できるようにする**（どのデータをいつ追加し、誰が確認したか）

**対策2：評価・監視・切り戻し（学習後）**

- 学習前後で性能を比較する
- 業務上重要なケース・攻撃されやすいケースでテストする
- 運用中は異常な出力を監視し、フィードバックやエラーログを活用する
- **学習データとモデルのバージョン管理** を行い、問題があれば前のモデルに戻せるようにする
- 重要な段階で人間が確認する運用にする

> **モデルだけでなく、学習データもセキュリティ管理の対象にする。** AIの安全性は「どのようなデータで学習したか」に大きく左右される。

## モデル改ざん・不正モデル・依存ライブラリのリスク

### 前提

AIアプリは、事前学習済みモデル、埋め込みモデル、トークナイザー、推論ライブラリ、RAGフレームワークなど多くの外部部品で構成されます。**モデルやライブラリも「外部から取り込むソフトウェア部品」として管理する** ことが基本です。

### 7つのリスク

| # | リスク | 概要 |
|---|--------|------|
| 1 | **モデル改ざん** | 正規のモデルに見せかけて中身が変更されたモデル。通常は正常に見えても、**特定の条件・入力でだけ不正な出力** をする可能性。配布元とファイルの正当性確認が必要 |
| 2 | **不正モデル** | 悪意ある目的で作成・改変・配布されたモデル。出所不明、第三者の再アップロード、公式を名乗る改変版、モデルカードなし、ライセンス不明、不審なスクリプトの実行を求めるものに注意 |
| 3 | **モデル読み込み時の危険** | モデルの保存形式・読み込み方法によっては、**読み込んだだけで不審なコードが実行される** 可能性。開発環境からのAPIキー窃取、外部への情報送信も。信頼できないモデルは隔離された検証環境で確認 |
| 4 | **依存ライブラリの脆弱性** | 直接使うライブラリだけでなく、その内部の依存関係（**間接依存・推移的依存**）の脆弱性にも注意。古いバージョンの使い続けも問題 |
| 5 | **悪意あるパッケージ** | 有名ライブラリに似た名前のパッケージ（**タイポスクワッティング**）、インストール時の不審なスクリプト実行、環境変数・秘密情報の外部送信。名前・提供元・更新履歴・利用実績・依存関係を確認 |
| 6 | **更新による挙動変化** | 出力形式の変化で後続処理が失敗、精度・エラー・セキュリティ制御の変化、互換性問題。**更新前に検証環境でテストし、戻せるようにしておく** |
| 7 | **秘密情報の混入** | APIキー・トークン・パスワード・接続文字列を、コード・モデル・学習データ・ログに含めない。専用の秘密情報管理機能を使い、漏洩時は速やかに無効化・再発行 |

### 安全な管理の基本

1. **信頼できる配布元から取得**（公式リポジトリ、提供元が案内する配布場所）
2. ハッシュ値・電子署名でファイルの改ざんがないか確認
3. ライセンスの確認（商用利用・改変・再配布・組み込み）
4. **バージョンを固定** し、どの環境で何を使っているか記録
5. 脆弱性スキャン、依存関係の棚卸し（不要なものは削除）
6. 更新前のテスト、更新後に問題があれば切り戻し

### 利用前チェックリスト

- [ ] 提供元が信頼できるか（公式版か、第三者の改変版でないか）
- [ ] モデル・ファイルが改ざんされていないか
- [ ] ライセンス条件に問題がないか
- [ ] モデルカードや説明資料があるか（用途・制限・評価結果・既知の問題）
- [ ] 既知の脆弱性がないか
- [ ] 不審な依存関係・不要なパッケージが含まれていないか
- [ ] 不要な権限や想定外の外部通信がないか
- [ ] 名前・配布元・バージョン・取得日時を記録したか
- [ ] 隔離された検証環境から利用を始めているか

## Model Denial of Serviceとコストリスク

### Model Denial of Serviceとは

**AIモデルに過大な負荷を与え、サービスを利用しにくくする攻撃・リスク** です。生成AIは1回の処理に多くの計算資源（CPU・GPU・メモリ・API利用枠）を使うため、大量のリクエストや高負荷な処理によって、応答遅延、処理待ち、サービス停止、利用枠の枯渇が起こり得ます。

### 通常のDoSとの違い

通常のDoSは大量の通信でサーバー資源を消費させますが、AIでは **少ないリクエストでも1件あたりの処理が重ければ大きな負荷になる** 点が特徴です。

- 長い入力・長い出力はトークン処理の負荷が大きい
- 推論処理がGPUやAPI利用枠を長時間占有する
- RAGでは文書検索・埋め込み生成も発生する
- エージェントではツール実行が連鎖する

**高負荷になりやすい入力**：非常に長い文章（契約書全文・大量のログ）、大量の文書・添付ファイル、毎回すべて送信される会話履歴、複雑な分析依頼、大量の候補生成・長い回答の要求、繰り返しの再生成、画像・音声・動画。

### AIエージェントで増えるリスク

ユーザーから見ると1回の依頼でも、内部では **多数のモデル呼び出しやツール実行** が発生し得ます（例：「競合調査レポート」→ 複数回のWeb検索 → 文書取得 → 要約・比較・分析 → 外部API → 長いレポート生成）。

**処理ループのリスク**：終了条件が不適切だと、同じ検索の繰り返し、エラー時の無制限な再試行、エージェント同士の呼び出し合い、回答の再生成の繰り返しが起こります。

→ **最大ステップ数、最大再試行回数、最大実行時間、最大ツール呼び出し回数** を設定し、超えた場合は処理を停止して人間に確認を求める。

### コストリスク

AIのコストはモデル利用料（入出力トークン）だけでなく、埋め込み生成・ベクトル検索、外部API呼び出し、GPU・クラウド・ストレージ、ログ・監視基盤にも発生します。**サービスが停止しなくても、予算超過は業務上の大きな問題** です。

**コストが急増しやすいケース**：

- 認証なしでAI機能をインターネットに公開している
- ユーザーごとの利用上限がない
- 出力トークン数を制限していない
- すべての処理で高性能モデルを使っている
- 長い会話履歴を毎回すべて送信している
- エラー時に無制限で再試行している
- 攻撃者による意図的な大量リクエスト

### 対策

**対策1：利用量を制限する**

| 制限項目 | 内容 |
|---------|------|
| ユーザー認証 | 誰が使っているか分からない状態で無制限に公開しない |
| レート制限 | 一定時間あたりのリクエスト回数を制限 |
| 同時実行数 | 同時に実行できる処理数を制限 |
| トークン上限 | 入力・出力トークン数に上限を設定 |
| ファイル制限 | アップロードのサイズ・数を制限 |
| タイムアウト | 長時間終了しない処理を停止 |
| 利用上限 | ユーザー・部署・契約プラン単位で設定 |

**対策2：処理とコストを制御する**

- 用途に合ったモデルを選ぶ（簡単な処理には軽量・低コストなモデル）
- 不要な会話履歴を毎回送らない
- RAGの検索件数を制限し、必要なチャンクだけ渡す
- キャッシュを活用する
- 再試行回数・最大ステップ数を制限する
- 高額な処理には人間の承認を入れる
- **異常な利用や予算超過時に自動で処理を停止できるようにする**

**監視すべき項目**：リクエスト数、同時実行数、入出力トークン数、処理時間、ツール実行回数、エラー・再試行回数、ユーザー別・部署別・モデル別の利用量、日次・月次のコストと予算に対する利用状況。

> **AIの利用量、処理範囲、予算に上限を設ける。** 可用性だけでなくコストも、AIセキュリティの重要な観点。

## セクション全体のまとめ

- **AIは単体で動いていない**。学習データ、モデル、ライブラリ、AI API、クラウド、RAG文書、ベクトルDB、外部ツールなどの依存先のどこかに問題があると、AIシステム全体に影響する。これがAIサプライチェーンの考え方
- サプライチェーン攻撃は、システムを直接ではなく **依存先や取り込み元を通じて** 狙う。データ混入、改ざんモデルの配布、悪意あるパッケージ、不審文書の取り込みなどが入口になる
- 外部モデル・APIは便利だが、データ送信、保存・学習利用、契約・ライセンス、挙動変化、サービス停止、ベンダーロックインのリスクがある。**出力の最終責任は利用側に残る**
- オープンソースモデルは「無料で使えること」と「安全に使えること」が別。ライセンス、出所、安全性、学習データの不透明性、運用責任を **自分たちで** 確認する
- **Training Data Poisoning** は学習データへの不正データ混入。学習前のデータ管理（出所・品質・履歴）と、学習後の評価・監視・切り戻しをセットで行う
- モデルやライブラリは **外部から取り込むソフトウェア部品** として管理する。信頼できる配布元、改ざん確認、バージョン固定、脆弱性管理、更新前テスト、秘密情報の分離が基本
- **Model Denial of Service** とコストリスクでは、長い入出力・処理ループ・エージェントの連鎖処理が負荷と費用を急増させる。利用量・処理範囲・予算に上限を設け、監視と自動停止の仕組みを用意する

### キーワード一覧

| 用語 | 意味 |
|------|------|
| AIサプライチェーン | AIを支えるデータ、モデル、ツール、サービス、運用のつながり |
| サプライチェーン攻撃 | 直接システムを攻撃せず、依存先や取り込み元を通じて影響を与える攻撃 |
| ベンダーロックイン | 特定の外部サービスに強く依存し、後から切り替えが難しくなる状態 |
| オープンソースモデル | 重み・コード・設定等が公開されたAIモデル。利用条件はモデルごとに異なる |
| モデルカード | モデルの用途、制限、評価結果、学習データの概要、ライセンス等を記した説明資料 |
| ファインチューニング | 既存モデルを自社データ等で追加調整すること。学習データの管理に注意が必要 |
| Training Data Poisoning | 学習データに不正なデータを混入させ、モデルの出力や判断に悪影響を与える攻撃・リスク |
| モデル改ざん | 正規のモデルに見せかけて中身が変更されたモデルを利用させるリスク |
| 間接依存（推移的依存） | 直接使うライブラリが内部で利用している、さらに先の依存関係 |
| タイポスクワッティング | 有名なライブラリに似た名前のパッケージで、入力ミスを狙って悪意あるコードを導入させる手法 |
| Model Denial of Service | AIモデルに過大な負荷を与え、サービスを利用しにくくする攻撃・リスク |
| レート制限 | 一定時間あたりのリクエスト回数を制限する仕組み |
| 処理ループ | エージェントが終了条件の不備により処理を必要以上に繰り返すこと。負荷と費用を消費する |
| バージョン固定 | モデルやライブラリのバージョンを固定し、更新時は検証してから反映する管理方法 |
